## ◈ What Lives Here

Semester 2 was the shift from *writing code* to *designing systems.* This repository is the complete archive of that transformation — every class hierarchy designed, every interface implemented, every runtime exception handled and survived.

Java was the language. **OOP was the discipline.** Everything here — from the first encapsulated class to the final campus project — was built with the intention of understanding *why* software is structured the way it is.

## ◈ The Four Pillars — Mastered
```
┌───────────────────────────────────────────────────────────────────┐
│                                                                   |
│   🔒  ENCAPSULATION      Private data. Public contracts.          |
│       Secure, modular design from the inside out.                 |
│                                                                   |
│   🧬  INHERITANCE       Build once.Extend everywhere.            |
│       Scalable class hierarchies that don't repeat themselves.    | 
│                                                                   |
│   🎭  POLYMORPHISM       One interface.Many forms.               |
│       Runtime flexibility baked into the architecture.            |
│                                                                   |
│   🌫️  ABSTRACTION        Hide complexity.Expose clarity.         |
│       Design what matters, bury what doesn't.                     |
│                                                                   |
└───────────────────────────────────────────────────────────────────┘
```

## ◈ Featured Projects

<details>
<summary><b>🚀 DeliveryBot — Logistics in Object Form</b></summary>
<br/>

A foundational project modeling a delivery system through OOP. Classes represent real entities — bots, orders, routes — teaching how to translate a real-world domain into working Java architecture.

**Concepts applied:** Class design, constructors, encapsulation, method interaction
</details>

<details>
<summary><b>📦 SIMS — Simple Inventory Management System</b></summary>
<br/>

Tracks and manages inventory items — adding, updating, removing, and displaying stock. An early exercise in thinking about *state* and how objects hold data that changes over time.

**Concepts applied:** Encapsulation, ArrayLists, CRUD logic within classes
</details>

<details>
<summary><b>🎓 SMS — Student Management System</b></summary>
<br/>

Handles student records — a clean, no-frills system that nails the basics of managing collections of objects, searching, and structured output.

**Concepts applied:** Object arrays, constructors, data display patterns
</details>

<details>
<summary><b>🚗 VMTS — Vehicle Management & Tracking System</b></summary>
<br/>

A blueprint for managing a fleet — vehicles as objects with properties and behaviors. A natural fit for **inheritance**: every vehicle shares a base, but each type has its own character.

**Concepts applied:** Inheritance, method overriding, polymorphism
</details>

<details>
<summary><b>📝 SEMS — Student Exam Management System</b></summary>
<br/>

Assigns and manages exams — juggling relationships between students, subjects, and scores. One of the more logic-dense projects requiring careful class design.

**Concepts applied:** Multi-class architecture, exception handling, relational object design
</details>

<details>
<summary><b>🏛️ UMS — University Management System</b></summary>
<br/>

Handles course registration at the university level. The scope here required thinking in *systems*, not just classes.

**Concepts applied:** Abstraction, interfaces, File I/O for persistence, exception handling
</details>

<details>
<summary><b>🏆 LabFinal — The Last Official Campus Project</b></summary>
<br/>

The capstone. Everything from the semester distilled into one final campus-affiliated submission. This one meant something.

**Concepts applied:** Full OOP stack — encapsulation, inheritance, polymorphism, abstraction, file handling, exception management
</details>

## ◈ Beyond the Four Pillars

**Exception Handling** — Runtime errors are inevitable. Learning to anticipate, catch, and recover from them gracefully is the difference between brittle code and robust software.

**File I/O & Persistence** — Data that vanishes when the program closes isn't useful. File reading and writing gave these systems *memory* — a crucial step toward real-world software behaviour.
